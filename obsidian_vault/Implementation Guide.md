This note explains how the current system is implemented and how to extend it safely.

## Current Implementation Principles

- Django REST Framework handles the core medical-record API.
- SimpleJWT provides token authentication.
- UUID primary keys are used across core models.
- Patients own records and structured clinical data.
- Sharing is implemented through expiring public tokens and doctor/patient grants.
- RAG is a separate FastAPI service and should be treated as a separate deployable.

## Adding a New Django Endpoint

Recommended steps:

1. Add or update the model in the relevant app.
2. Create a migration.
3. Add a serializer with explicit fields.
4. Add a view or viewset.
5. Add a permission class for role/object access.
6. Wire the route in the app `urls.py`.
7. Add tests for allowed and denied access.
8. Update [[API Endpoints]] and related vault notes.

Example structure:

```python
class IsPatientOwner(BasePermission):
    def has_object_permission(self, request, view, obj):
        return obj.user_id == request.user.id
```

Avoid:

- Relying on frontend checks for authorization.
- Returning `fields = "__all__"` for sensitive objects.
- Using `ModelViewSet` when only read or custom actions are intended.

## Adding a New Model

Checklist:

- Use UUID primary keys for consistency.
- Add `created_at` and `updated_at` where lifecycle matters.
- Add indexes for common filters, especially `user`, `patient`, `doctor`, dates, and status fields.
- Use enumerated choices for statuses.
- Add database constraints for uniqueness and invariant rules.
- Add admin registration only if useful for local/admin operations.
- Add tests for model constraints and access patterns.

For PHI-related models, also add:

- Audit event when created/read/updated/deleted.
- Clear ownership semantics.
- Retention/deletion policy.

## Record Upload Implementation

Current flow in `records/views.py`:

1. Validate metadata with `RecordUploadSerializer`.
2. Read uploaded file from `request.FILES["file_url"]`.
3. Check client-provided content type.
4. Compute SHA-256 hash by iterating `uploaded_file.chunks()`.
5. Reset file pointer with `uploaded_file.seek(0)`.
6. Save `Record` and `BlockchainTransaction` in `transaction.atomic()`.
7. Start background blockchain simulation thread.
8. Return record metadata to caller.

Implementation details to preserve:

- Hash before saving.
- Use database transaction for record and transaction row.
- Return `PENDING` initially; async status changes later.

Implementation details to improve:

- Validate actual file signature.
- Add size limits.
- Store uploads privately.
- Queue blockchain work durably.
- Audit upload event.

## Sharing Implementation

Current token sharing:

- `ShareToken.token` is generated by `secrets.token_urlsafe(32)`.
- `ShareToken.record` null means full-vault sharing.
- `expires_at` defaults to one hour from creation.
- The public endpoint validates token existence and expiry.

Current grant sharing:

- A doctor creates an `AccessRequest`.
- The patient approves it.
- Approval creates an `AccessGrant`.
- Doctor endpoints check that a grant exists.

Implementation details to improve:

- Store hashed tokens.
- Do not log tokens.
- Add token revocation.
- Enforce doctor role when creating requests.
- Make `AccessGrant.records` semantics real or remove it.
- Add audit events for token generation, token access, request approval, grant reads, and revocation.

## Appointment Implementation

Current flow:

- `AppointmentViewSet.get_queryset()` scopes appointments by current user role.
- Patients see appointments where `user=request.user`.
- Doctors see appointments by doctor FK or free-form name fallback.
- `perform_create()` tries to find a doctor by parsed name.
- `confirm` and `cancel` actions check patient/doctor relationship.

Implementation details to improve:

- Require `doctor_id` when booking.
- Keep `doctor_name` only as a denormalized display snapshot.
- Add conflict checks.
- Replace free-form `status` with choices.
- Add tests for authorization and doctor-name collision cases.

## RAG Implementation

Current flow:

1. `POST /api/v1/reindex` builds documents from DB and patient JSON files.
2. Documents are chunked.
3. Chunks are embedded.
4. FAISS index and metadata are saved.
5. `POST /api/v1/query` validates JWT.
6. Patient role is required.
7. Retrieval is scoped to JWT `user_id`.
8. LLM generates answer with sources.

Implementation details to improve:

- Make reindex admin/service-only.
- Align RAG DB connector with Django production database.
- Add index locking and versioning.
- Prevent arbitrary patient compare.
- Avoid logging PHI queries.

## Error Handling Pattern

Current code uses a mix of:

- serializer validation errors
- explicit `Response(..., status=...)`
- broad `except Exception`
- FastAPI `HTTPException`

Production target:

- Avoid broad exception swallowing.
- Log internal error details server-side only.
- Return stable client-safe error messages.
- Add request IDs to logs and responses.
- Keep PHI and tokens out of error messages.

## Settings Implementation Plan

Recommended future settings layout:

```text
medchain_backend/settings/
|-- base.py
|-- local.py
|-- test.py
|-- production.py
```

`base.py`:

- Installed apps.
- Middleware.
- REST framework defaults.
- Common auth settings.

`local.py`:

- Debug true.
- Local database.
- Local media.
- Console email/logging.

`production.py`:

- Debug false.
- Secure cookies.
- HSTS.
- Object storage.
- Strict CORS/hosts.
- Secret-manager env variables required.
